🍽️ Yelp 商家智能问答 Agent

基于 RAG 的领域问答系统：三种架构 × 多种部署后端的对比研究 · 2026 年 4 月 – 5 月

在线 Demo： huggingface.co/spaces/YUHAN03/yelp-rag-agent
技术栈： FAISS sentence-transformers LangGraph ReAct LangChain Gradio 5 Qwen2.5-7B（研究） Llama 3.1 8B / Groq（Demo）
数据规模： 60,823 条评论 chunk · 50,000 条原始评论 · 160 家商家

1. 项目目标

在 Yelp 评论领域构建问答系统，并严肃地回答两个工程问题：

  1. 质量问题：纯 LLM、固定 RAG 流水线、自主 Agent 三种架构里， 哪种在领域问答中表现最好？
  2. 部署问题：在单卡 A100 80GB 上，AWQ（W4A16 量化） 通过 LMDeploy 部署相比 fp16 baseline 能提速多少？质量是否退步？

2. 系统架构

用户问题
   │
   ├─► Direct LLM        ── Qwen2.5-7B / Llama 3.1，仅依赖参数化记忆
   │
   ├─► RAG Baseline      ── Stats → FAISS 检索 → 结构化合成

                            classifier 等工具

2.1 检索栈

  • Chunk 级 FAISS 索引（IndexFlatIP），共 60,823 条 chunk。 评论按 180 词、句边界感知地切分；embedding 经 L2 归一化， 使内积等价于余弦相似度。
  • 商家预过滤检索：维护 business_to_indices 映射， 直接在目标商家的 embedding 子集上算相似度，避免"全局 Top-K + 后过滤" 对低评论量商家失效的反模式。
  • Embedding 模型：sentence-transformers/all-MiniLM-L6-v2（384 维）。

2.2 后端抽象层（核心工程思路）

所有 pipeline 只依赖 BaseBackend 接口的一个方法 generate(prompt, temperature, max_tokens) -> str。 四个具体后端通过 YAML 配置即可互换：

后端用途调用端点
OllamaBackend 本地开发 localhost:11434
LMDeployBackend A100 部署研究 OpenAI 兼容 HTTP
GroqBackend HF Spaces 在线 demo（现役）api.groq.com/openai/v1
HFInferenceBackend备用 serverless 方案 api-inference.huggingface.co

切换后端不需要改任何应用层代码 — 只需 CLI 参数 （--config configs/groq.yaml）或 HF Space 上的一条 secret。

3. 研究结果 — A100 部署研究（Qwen2.5-7B）

3.1 部署性能

在冷启动的 NVIDIA A100 80GB GPU 上，通过 LMDeploy 0.7.x 测量。

指标 fp16 (pytorch) AWQ (turbomind) AWQ 相对优势
TTFT（首 token 时延）（毫秒）693.618.038.5× 更快
吞吐（tokens/秒） 73.9 166.92.26× 更高
平均响应时间（秒） 2.00 0.912.20× 更快
VRAM 峰值（GB） 64.5565.30基本持平*

*VRAM 看起来基本一致，原因是 LMDeploy 的 turbomind 后端在启动时预分配了 一大块 KV-cache 缓冲区，占了总占用的大头。在显存受限的 GPU（如 T4 16GB）上， AWQ 的 W4A16 权重压缩才会真正发挥优势。

3.2 答案质量（3 系统 × 20 题 · 人工 0–2 分 / 维度评分，单题满分 10）

系统 后端 正确性证据 可溯源工具使用 效率总分 /10
Direct LLMA100 / Qwen 0.200.000.000.000.951.15
RAG BaselineA100 / Qwen 1.751.901.952.001.909.50
Full AgentA100 / Qwen 0.000.000.001.000.001.00
Full Agent（修复后）Groq / Llama 3.1 8B 1.801.301.251.851.407.60
Full Agent 的“修复”— 一次后端切换带来 7.6× 质量回升。上表中划掉的那一行来自原 A100 研究（Qwen2.5-7B），它输出 非标准的 <tool_call> XML 标签，LangChain 直接静默丢弃。 把 LLM 后端切换到 Groq 上的 Llama 3.1 8B Instant（输出标准 tool_calls JSON）后，Full Agent 总分从 1.00 跳到 7.60 / 10 — pipeline 和 tool 代码零修改，只新增了约 30 行 GroqBackend 封装类和一份 YAML 配置。20 题 Llama 评测里 幻觉率为 0%（没有一题 groundedness 为 0）。

即使在修复后，Full Agent（7.60）在 Evidence（1.30 vs 1.90）和 Groundedness（1.25 vs 1.95）维度上仍略低于 RAG Baseline（9.50）。这一差距 来源于架构本身：RAG Baseline 的固定流水线强制走 summarize_evidence 步骤，输出结构化的 claim + quote 块；而 Full Agent 让 LLM 自由合成 答案 — 更自然但显式引用少一些。

5. 工程挑战与解决方案

5.1 Qwen2.5 tool-calling 兼容性问题（研究阶段发现）

在 A100 研究中，Full Agent 系统得分接近 0，尽管模型正确识别出了要调用的工具。 根因：Qwen2.5-7B 输出 <tool_call>...</tool_call> XML 标签，而非 OpenAI 标准的 tool_calls JSON，LangChain 的 ChatOpenAI 客户端无法解析。这是框架集成限制，不是 模型能力限制 — 在 benchmark 报告中有完整记录。

在线 Demo 的修复方案：把 LLM 后端切换到 Groq 上的 Llama 3.1 8B Instant，它输出标准 OpenAI tool_calls。这次切换没有 改动任何 pipeline / tool 代码，只新增了一个 GroqBackend 类（约 30 行）和一份 YAML 配置。这正是后端抽象层的工程价值兑现。

5.2 懒加载单例的多线程竞态

HF 首次部署时报错 AttributeError: 'NoneType' object has no attribute 'encode'， 在 Full Agent 并行调用多个工具时触发。 根因：LangGraph 把 tool 调用派发到线程池；_load_store() 懒加载 函数不是线程安全的。三个线程同时调用 SentenceTransformer(...).to('cuda') 破坏了 PyTorch 内部 CUDA 状态，导致某些线程视角下 _embed_model 是 None。

修复：在所有懒加载路径 （retrieval_tool / stats_tool / classifier_tool）加 threading.Lock + 双检锁。

5.3 Groq 对 tool 消息的严格 schema 校验

Groq 的 chat completion API 会拒收 content 为空 list（content: []） 的 tool 消息，返回 HTTP 400。当 LLM 偶尔用错的 business_id 调用 search_review_chunks_by_business 时，工具返回 []， 下一轮请求就被拒。

修复：检索工具改为返回 JSON 字符串（空结果时返回 "No matching review chunks were found..." 这种提示字符串）。 RAG Baseline 在自己侧把 JSON 字符串解析回 list。

5.4 Agent 工具集精简

Llama 3.1 8B 偶尔会陷入循环或多余调用 classify_review。 务实修复：从 Agent 工具集移除 summarize_evidence 和 classify_review（检索结果本身已包含星级数据，无需再分类）。 最终 Agent 工具栈：search_global、 search_by_business、get_business_stats。

6.5 架构远见

后端抽象层最初是为了一个研究上的便利而设计的 — 同一份代码 既要能跑在 Ollama 上（本地开发），又要能跑在 LMDeploy 上（A100 部署研究）。 它不是为了应付框架层 tool-calling bug 而设计的。但几个月后 当 Qwen ↔ LangChain 不兼容问题浮现时，这个抽象层让修复成本变成约 30 行新代码，而不是一次重写。 

当初设计是为了：   A100 部署研究 ──────────────┐
                                              ├──►  后端抽象层
意外救了：         Qwen tool-calling 解析 bug ─┘                │
                  (Full Agent → 1.00 / 10)                     │
                                                                ▼
                                                  30 行的 GroqBackend 封装
                                                                │
                                                                ▼
                                                  Full Agent → 7.60 / 10（7.6×）

一个为某目的设计的抽象，意外回报了一个完全不同的场景。这种回报只有在抽象 第一次就设计对的时候才可能发生 — bug 浮现后再回头加抽象，就 意味着把每个 pipeline 都重写一遍。

7. 核心结论

  • RAG 比 Direct LLM 高出 8 倍质量分，即使用 7B 小模型亦然 — 在领域问答中检索 grounding 才是质量的主导因素，而不是模型尺寸。
  • AWQ 在 A100 上相比 fp16 实现 38.5× 的 TTFT 降低 + 2.26× 的吞吐提升 且无可测的质量损失 — 单卡 A100 生产部署应默认使用 AWQ + turbomind。
  • 后端抽象层在真实迁移中确实回本：把 LMDeploy/Qwen 切换到 Groq/Llama 做免费 serverless demo，只需约 30 行代码加 1 份 YAML。
  • 框架集成 bug 和模型质量是两件事：«Agent 在 Qwen 上不工作» 是 parser 问题，不是推理问题。切换到 Groq 的 Llama 3.1 8B 后，Full Agent 总分从 1.00 升到 7.60 / 10（7.6×），且应用层代码 一行未动 — 后端抽象层的工程红利在这一次迁移中就回本了。
