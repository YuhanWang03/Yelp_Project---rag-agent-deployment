🍽️ Yelp Business Intelligence Agent

A RAG-powered question-answering system comparing three architectures across deployment backends · April – May 2026

Live demo: huggingface.co/spaces/YUHAN03/yelp-rag-agent
Stack: FAISS sentence-transformers LangGraph ReAct LangChain Gradio 5 Qwen2.5-7B (research) Llama 3.1 8B / Groq (demo)
Data: 60,823 review chunks · 50,000 sampled Yelp reviews · 160 businesses

1. Project Goal

Build a domain-specific question-answering system on Yelp customer reviews, and rigorously answer two engineering questions:

  1. Quality: Direct LLM vs fixed RAG pipeline vs autonomous agent — which architecture answers domain questions best?
  2. Deployment: On a single A100 80GB, how much speedup does AWQ (W4A16 quantization) give over fp16 baseline via LMDeploy, and is there any quality regression?

2.1 Retrieval stack

  • Chunk-level FAISS index (IndexFlatIP) over 60,823 chunks. Reviews chunked at 180 words with sentence-aware splitting; embeddings L2-normalised so inner product = cosine similarity.
  • Pre-filtered business search: a business_to_indices map enables direct retrieval on a single business's embedding subset, avoiding the global Top-K + post-filter failure mode for small-volume businesses.
  • Embeddings: sentence-transformers/all-MiniLM-L6-v2 (384-dim).

2.2 Backend abstraction layer (key engineering thesis)

All pipelines depend only on a BaseBackend interface with one method, generate(prompt, temperature, max_tokens) -> str. Four concrete backends are interchangeable via YAML config:

BackendUse caseEndpoint
OllamaBackend Local development localhost:11434
LMDeployBackend A100 deployment study OpenAI-compatible HTTP
GroqBackend HF Spaces live demo (current)api.groq.com/openai/v1
HFInferenceBackendFallback serverless option api-inference.huggingface.co

Switching backends requires no application code change — only a CLI flag (--config configs/groq.yaml) or one Space secret.

The Full Agent “fix” — a 7.6× quality recovery from one backend swap. The crossed-out row above is from the original A100 study with Qwen2.5-7B, which emits non-standard <tool_call> XML tags that LangChain silently drops. Switching the LLM backend to Llama 3.1 8B Instant on Groq (standard tool_calls JSON) lifted the Full Agent from 1.00 to 7.60 / 10 — with zero changes to pipeline or tool code, only ~30 lines of new GroqBackend wrapper plus a YAML config. 0% hallucination rate in the 20-question Llama evaluation (groundedness = 0 on no question).

Even after the fix, Full Agent (7.60) is slightly below RAG Baseline (9.50) on Evidence (1.30 vs 1.90) and Groundedness (1.25 vs 1.95). The gap is explained by architecture: RAG Baseline's fixed pipeline forces an evidence-synthesis step that emits structured claim + quote blocks, while the Full Agent has the LLM compose answers freely — more natural reading, less explicit citation.

5. Engineering Challenges & Solutions

5.1 Qwen2.5 tool-calling incompatibility (research finding)

During the A100 study, the Full Agent system scored near-zero across all dimensions despite the model correctly identifying tools to call. Root cause: Qwen2.5-7B emits <tool_call>...</tool_call> XML tags rather than OpenAI's standard tool_calls JSON, which LangChain's ChatOpenAI client cannot parse. This is a framework integration limitation, not a model capability limitation — documented in the benchmark report.

Resolution for the live demo: Swapped the LLM backend to Llama 3.1 8B Instant via Groq, which emits standard OpenAI tool_calls. Required zero changes to pipeline / tool code — only one new GroqBackend class (≈30 lines) and a YAML file. This is the engineering payoff of the backend abstraction layer.

5.2 Race condition in lazy singleton initialisation

Initial HF deployment threw AttributeError: 'NoneType' object has no attribute 'encode' when the Full Agent fired multiple tool calls in parallel. Root cause: LangGraph dispatches tool calls to a thread pool; the _load_store() lazy initialiser was not thread-safe. Three threads simultaneously called SentenceTransformer(...).to('cuda'), corrupting PyTorch's internal CUDA state and leaving _embed_model as None in some threads' view.

Fix: Added threading.Lock with double-checked locking around all lazy-init paths (retrieval_tool, stats_tool, classifier_tool).

5.3 Groq strict tool-message schema

Groq's chat completion API rejects tool messages whose content is an empty list (content: []) with HTTP 400. When the LLM accidentally called search_review_chunks_by_business with a bad business_id, the tool returned [] and Groq refused the follow-up request.

Fix: Search tools now return JSON-serialised strings (or a sentinel "No matching review chunks were found..." string on empty results). RAG Baseline parses the JSON string back into a list.

5.4 Agent tool-set minimisation

Llama 3.1 8B occasionally got stuck in loops or made redundant classify_review calls. Pragmatic fix: removed summarize_evidence and classify_review from the agent's toolset (search results already include star ratings in context). Final agent tool surface: search_global, search_by_business, get_business_stats.

7. Take-aways

  • RAG > Direct LLM by 8× on quality for domain QA, even with a 7B model — retrieval grounding is the dominant factor, not model size.
  • AWQ delivers 38.5× lower TTFT and 2.26× higher throughput over fp16 on A100 with no measurable quality loss — production deployments on a single A100 should default to AWQ + turbomind.
  • Backend abstraction pays off in real migrations: swapping LMDeploy/Qwen → Groq/Llama for a free serverless demo required ~30 lines of code and one YAML.
  • Framework integration bugs are real and orthogonal to model quality: the «agent doesn't work on Qwen» finding is a parser issue, not a reasoning issue. Swapping to Llama 3.1 8B on Groq raised the Full Agent score from 1.00 to 7.60 / 10 (7.6×) with zero application-layer changes — the backend abstraction layer paid for itself in this single migration.
